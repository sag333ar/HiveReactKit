# Content translation (i18n)

Translate Hive post bodies, comments, titles, snap captions, and feed previews into the user's chosen language without touching every component. Wrap your app once with `<HiveLanguageProvider>` and every body / title rendered by hive-react-kit follows that language.

The kit ships a free **MyMemory**-backed translator by default — no API key, soft daily quota, no opinion. Override it with DeepL / Google / your own backend via a single prop.

## At a glance

```tsx
import { HiveLanguageProvider } from 'hive-react-kit'
import { useTranslation } from 'react-i18next' // your existing app i18n

function App() {
  const { i18n } = useTranslation()
  const language = i18n.resolvedLanguage || i18n.language || 'en'

  return (
    <HiveLanguageProvider language={language}>
      {/* …rest of your app… */}
    </HiveLanguageProvider>
  )
}
```

That's the whole integration. When `language === 'en'` the provider is a no-op — no API calls, no overhead. When `language === 'es'` (or any other supported MyMemory pair), every kit-rendered Hive content body translates in place, cached for instant re-render on revisit.

## What gets translated automatically

The provider drives **two** kinds of translation:

### User-generated content (post bodies, titles, captions)

Translated through MyMemory (or your custom translator) at runtime, cached in localStorage.

| Component | What translates |
|-----------|-----------------|
| `HiveDetailPost` | Post title + post body |
| `InlineCommentItem` | Comment body |
| `CommentTile` | Both render branches (search-highlighted + plain) |
| `UserChannel` | Activity feed body |
| `UserDetailProfile` (feed cards) | Each item's title + body preview — covers blogs / posts / snaps / comments / replies / rewards tabs |
| `PollListItem` | Poll question, body preview, every choice label |

The composer preview inside `AddCommentInput` is **not** translated — that's the user's own draft, you don't want their typing translated mid-sentence.

### Kit UI labels (tabs, buttons, status pills, modals)

Resolved instantly from the built-in messages dict (zero network calls). Out of the box `en` and `es` are provided; consumers can extend via the `messages` prop.

| Where | Examples |
|-------|----------|
| `UserDetailProfile` tab bar | "Blogs" / "Publicaciones", "Voting Power" / "Poder de voto", … (16 tabs) |
| `UserDetailProfile` profile meta | "Followers", "Following", "Posts" stats footer |
| `UserDetailProfile` action menu | "Follow" / "Seguir", "Unfollow" / "Dejar de seguir", "Ignore Author", "Report User", "Share Profile" |
| `UserDetailProfile` ignore-confirm modal | Title, body (with `@username` interpolation), Cancel, Confirm Ignore, Processing |
| `UserDetailProfile` empty states | "No posts found", "Not following anyone", "No badges found", and 11 more |
| `UserDetailProfile` rewards summary | "Pending Author Rewards", "Total HBD", "Total HP", "Avg Efficiency", "Post"/"Comment" row tags |
| `UserDetailProfile` voting power | "Voting Power", "Downvote Power", "Resource Credits" labels + "Fully charged" |
| `PollListItem` | "Active" / "Activa" pill, "Submit Vote" / "Enviar voto", "Change Vote", "Voted", "Vote changes allowed", voter / option pluralisation, "Ends {when}" |

## API

### `<HiveLanguageProvider>`

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `language` | `string` | `'en'` | Two-letter language code. `'en'` is the no-op pass-through. Any code MyMemory accepts (`'es'`, `'fr'`, `'de'`, `'pt'`, `'it'`, `'ja'`, `'zh'`, `'ko'`, `'ar'`, `'hi'`, `'ru'`, `'tr'`, `'nl'`, `'pl'`, `'sv'`, …) works out of the box. Drives both content translation and UI label lookup. |
| `translateHtml` | `(html: string, target: string) => Promise<string>` | bundled MyMemory implementation | Override translator. Receives the rendered HTML and target language; should return translated HTML. Useful for plugging in DeepL, Google Translate, an authenticated MyMemory account, or your own API. |
| `messages` | `Record<string, Record<string, string>>` | bundled `en` + `es` dicts | Override / extend kit UI labels. Key is the language code, value is a flat `key → label` dict. Merges on top of the built-ins; missing keys fall back to English. Use this to add a third language or rewrite specific labels (e.g. localise "Follow" to your brand voice). |
| `children` | `ReactNode` | *required* | App tree. |

### `useHiveLanguage()`

```ts
const { language, translateHtml } = useHiveLanguage()
```

Read the current language + translator from anywhere in the tree. Mostly useful when you want to translate a string yourself — most consumers should use the wrapper components below.

### `useKitT()`

```tsx
const t = useKitT()
t('action.follow')                          // "Follow" / "Seguir"
t('modal.ignoreAuthorBody', { username })   // "Are you sure you want to ignore @alice…"
```

Returns the synchronous UI-label resolver. Only useful when you're building a custom component that should match the kit's translation behaviour — every kit-shipped component already calls this internally.

Resolution order: consumer override (your `messages` prop) → kit built-ins for the active language → consumer override for English → kit built-ins for English → the key itself.

### `useTranslatedHtml(html)`

```tsx
const { html, loading } = useTranslatedHtml(rawHtml)
```

Hook for translating rendered HTML (output of `createHiveRenderer` / `renderHiveMarkdown`). Returns the original synchronously while the translation request is in flight, then swaps to the translated version. Cancels stale promises on unmount and on language change.

Use this if you have your own custom rendering pipeline that runs after `createHiveRenderer` and you can't drop in `<TranslatedBody>` (e.g. you do post-mount DOM surgery on the HTML, like 3Speak / YouTube embed swaps).

### `useTranslatedText(text)`

```tsx
const { text, loading } = useTranslatedText(post.title)
```

Plain-text counterpart of `useTranslatedHtml`. Use for titles, captions, preview snippets — any short user-generated string. Same caching layer is shared, so the title and body of the same post don't make duplicate API calls.

### `<TranslatedBody>`

Drop-in replacement for the `<div … dangerouslySetInnerHTML>` pattern.

```tsx
<TranslatedBody
  className="hive-post-body max-w-none"
  html={renderedBody}
  onClick={handleBodyClick}
/>
```

Forwards refs to the inner div so existing click delegation / image fallback / observer logic continues to attach correctly:

```tsx
const bodyRef = useRef<HTMLDivElement>(null)
useImageFallback(bodyRef)

<TranslatedBody ref={bodyRef} className="hive-post-body" html={html} />
```

While translation is in flight the original HTML is shown and `data-translating="true"` is set on the rendered div, so consumers can apply a CSS-only loading style if they want:

```css
[data-translating="true"] {
  opacity: 0.7;
  transition: opacity 200ms ease;
}
```

### `<TranslatedText>`

Inline wrapper for plain strings — the rules-of-hooks-safe way to translate text inside `.map()` render loops:

```tsx
{posts.map((post) => (
  <article key={post.permlink}>
    <h2><TranslatedText text={post.title} /></h2>
    <p><TranslatedText text={post.preview} /></p>
  </article>
))}
```

Returns the original synchronously, swaps to translated when ready, no-op for English.

## Customising UI labels

The kit ships built-in `en` and `es` dicts covering every visible label in `UserDetailProfile`, `PollListItem`, and the shared modals. To override a specific label or add a new language, pass `messages` to the provider:

```tsx
<HiveLanguageProvider
  language="fr"
  messages={{
    fr: {
      "action.follow": "Suivre",
      "action.unfollow": "Ne plus suivre",
      "tab.blogs": "Blogs",
      "tab.posts": "Publications",
      // …only the keys you care about
    },
    es: {
      // Override one Spanish label without losing the rest of the bundled es dict
      "action.follow": "Seguir cuenta",
    },
  }}
>
  <App />
</HiveLanguageProvider>
```

Missing keys for a non-English language fall back to the built-in English label, so you can ship a partial dict and still have a working UI.

The full key list is exported as the `KitMessageKey` type:

```ts
import type { KitMessageKey } from 'hive-react-kit'

const myMessages: Partial<Record<KitMessageKey, string>> = {
  'action.follow': '…',
}
```

Categories: `tab.*`, `meta.*`, `action.*`, `modal.*`, `empty.*`, `status.*`, `poll.*`, `common.*`, `reward.*`, `vp.*`. The default English values are the source of truth — see [`src/i18n/messages.ts`](../src/i18n/messages.ts) in the kit for the canonical list.

### Placeholders

Templates use `{name}` placeholders interpolated at call time:

```ts
'modal.ignoreAuthorBody':
  'Are you sure you want to ignore @{username}? Their content will be hidden from your feed.'
```

Pass values as the second `t()` argument (or in your `messages` override) and they're substituted directly:

```tsx
t('modal.ignoreAuthorBody', { username: 'alice' })
// → "Are you sure you want to ignore @alice? Their content will be hidden from your feed."

t('poll.selected', { count: 3 })
// → "3 selected" / "3 seleccionada(s)"
```

## Custom translator

The kit's bundled MyMemory translator is fine for free-tier use. For production / higher volume / better quality, override at the provider:

```tsx
import { HiveLanguageProvider, type TranslateHtmlFn } from 'hive-react-kit'

const translateWithDeepL: TranslateHtmlFn = async (html, target) => {
  const res = await fetch('/api/translate', {
    method: 'POST',
    body: JSON.stringify({ html, target }),
    headers: { 'Content-Type': 'application/json' },
  })
  const { translated } = await res.json()
  return translated
}

<HiveLanguageProvider language={language} translateHtml={translateWithDeepL}>
  <App />
</HiveLanguageProvider>
```

Your function is called with the **already-rendered HTML** of each body (post-`createHiveRenderer`). It should return translated HTML. The kit handles caching at the provider level — if you want server-side caching too, add it transparently in your endpoint.

If you only need plain-string translation (no HTML walking), the kit's hooks call `translateText` from `@/i18n` directly — that uses MyMemory regardless of your `translateHtml` override. To override plain-text translation, fork or replace the service module.

## Caching

The default translator caches three ways:

1. **In-memory `Map`** — session-scoped, hit on every re-render.
2. **localStorage** under `hive-react-kit-translation-cache` — survives reloads. LRU-pruned at 500 entries (drops oldest 100 when full).
3. **In-flight dedup** — concurrent renders of the same body issue one network request; the rest await the same Promise.

Cache key is the hash of `(sourceLang | targetLang | text/html)` so the same body across multiple list items / re-renders / detail views collapses to one fetch. Title + body of the same post share the cache, so they don't double-bill against the API quota.

Custom translators are not auto-cached by the kit — wire your own LRU if needed.

## Translation behaviour

- **What the HTML walker translates:** every visible text node.
- **What it skips:** text inside `<code>`, `<pre>`, `<script>`, `<style>`, `<noscript>`, `<iframe>` — code blocks render unchanged, embeds (3Speak, YouTube, X) keep their src attributes intact.
- **What's preserved:** all attributes (links, image alt, image src, classes), markup structure, embeds, custom elements.
- **Text length:** strings over 480 chars are chunked on sentence boundaries before requesting (MyMemory's anonymous endpoint caps at 500). Translations are stitched back together with a single space.

## Failure modes

- **Network error / API rate limit:** the original text/HTML is returned. UI never breaks.
- **Quota exceeded:** MyMemory returns warning sentinels (`MYMEMORY WARNING`, `QUERY LENGTH LIMIT EXCEEDED`); the translator detects these and falls back to the original.
- **Empty input:** returned unchanged.

## Example: hooking into `react-i18next`

```tsx
// main.tsx
import './i18n'  // your existing i18next init

// App.tsx
import { useTranslation } from 'react-i18next'
import { HiveLanguageProvider } from 'hive-react-kit'

export default function App() {
  const { i18n } = useTranslation()
  const language = i18n.resolvedLanguage || i18n.language || 'en'

  return (
    <HiveLanguageProvider language={language}>
      <Routes>{/* … */}</Routes>
    </HiveLanguageProvider>
  )
}
```

Switching the user's language (e.g. via `i18n.changeLanguage('es')`) re-renders the provider with the new code, every translated component picks it up, and the localStorage cache is consulted before any API call goes out.

## Example: custom integration without a top-level provider

If your app doesn't use react-i18next or any global state, you can drive language manually:

```tsx
const [lang, setLang] = useState<'en' | 'es'>('en')

return (
  <>
    <button onClick={() => setLang(lang === 'en' ? 'es' : 'en')}>
      Toggle: {lang}
    </button>
    <HiveLanguageProvider language={lang}>
      <HiveDetailPost author="alice" permlink="hello-world" />
    </HiveLanguageProvider>
  </>
)
```
